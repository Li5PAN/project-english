# 英语学习系统 - 开发方法与开发工具

## 项目概述

本项目采用前后端分离的开发模式，遵循敏捷开发方法论，使用面向对象的分析与设计思想，通过专业的开发工具和辅助工具完成系统的设计、开发、测试和部署。

---

## 一、开发方法

### 1.1 面向对象分析与设计（OOA/OOD）

**方法论说明：**

面向对象分析与设计是一种软件工程方法，通过识别系统中的对象、类、属性和方法，以及它们之间的关系，来构建软件系统。本项目严格遵循面向对象的三大特性（封装、继承、多态）和五大原则（SOLID 原则）进行系统设计。

**在本项目中的应用：**

#### 1.1.1 需求分析阶段（OOA）

**用例分析：**
- 识别系统的三类主要参与者：学生（Student）、教师（Teacher）、管理员（Admin）
- 分析每个角色的核心用例：
  - 学生：登录系统、学习单词、查看任务、提交学习记录、申请换班、查看学习进度
  - 教师：管理班级、发布任务、审核换班申请、查看学生数据、分析班级统计
  - 管理员：管理用户、管理班级、系统配置、数据维护

**领域模型建立：**

识别核心业务实体及其属性和关系：

```
├─────────────────┤

        △

   │         │        │
```

**关系分析：**
- User 与 Class：多对一关系（一个班级有多个学生）
- Teacher 与 Class：一对多关系（一个教师管理多个班级）
- Task 与 Class：多对一关系（一个班级有多个任务）
- LearningRecord 与 User：多对一关系（一个用户有多条学习记录）
- LearningRecord 与 Word：多对一关系（一个单词被多次学习）


#### 1.1.2 系统设计阶段（OOD）

**分层架构设计：**

采用经典的三层架构模式，实现关注点分离：

```
┌─────────────────────────────────────────────┐
│          表示层（Presentation Layer）         │
│  - 前端：Vue 3 组件                           │
│  - 后端：Controller（接收请求、返回响应）      │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│          业务逻辑层（Business Layer）         │
│  - Service 接口：定义业务方法                 │
│  - ServiceImpl 实现：具体业务逻辑             │
│  - 事务管理、权限校验、业务规则验证            │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│          数据访问层（Data Access Layer）      │
│  - Mapper 接口：定义数据操作方法              │
│  - MyBatis XML：SQL 映射文件                 │
│  - 实体类（Entity）：对应数据库表             │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│              数据库（MySQL）                  │
└─────────────────────────────────────────────┘
```

**设计模式应用：**

1. **单例模式（Singleton Pattern）**
   - 应用场景：Spring Bean 默认为单例模式
   - 示例：UserService、WordService 等服务类

2. **工厂模式（Factory Pattern）**
   - 应用场景：Spring IoC 容器作为对象工厂
   - 示例：通过 @Autowired 注入对象实例

3. **代理模式（Proxy Pattern）**
   - 应用场景：Spring AOP 实现权限控制、事务管理
   - 示例：@Transactional 注解、@DataScope 自定义注解

4. **策略模式（Strategy Pattern）**
   - 应用场景：不同角色的数据权限过滤策略
   - 示例：学生、教师、管理员的数据访问策略

5. **模板方法模式（Template Method Pattern）**
   - 应用场景：统一的 CRUD 操作流程
   - 示例：BaseService 定义通用增删改查模板


**类的设计原则（SOLID）：**

1. **单一职责原则（Single Responsibility Principle）**
   - 每个类只负责一个功能领域
   - 示例：UserController 只处理用户相关请求，UserService 只处理用户业务逻辑

2. **开闭原则（Open-Closed Principle）**
   - 对扩展开放，对修改关闭
   - 示例：通过接口定义规范，新增功能时实现新接口而不修改原有代码

3. **里氏替换原则（Liskov Substitution Principle）**
   - 子类可以替换父类
   - 示例：Student、Teacher、Admin 都可以作为 User 使用

4. **接口隔离原则（Interface Segregation Principle）**
   - 客户端不应依赖它不需要的接口
   - 示例：将 Service 拆分为多个细粒度接口

5. **依赖倒置原则（Dependency Inversion Principle）**
   - 依赖抽象而非具体实现
   - 示例：Controller 依赖 Service 接口，而非具体实现类

### 1.2 敏捷开发方法（Agile Development）

**方法论说明：**

采用敏捷开发的迭代式开发模式，将整个项目划分为多个迭代周期（Sprint），每个迭代完成一部分功能，快速交付可用的软件增量。

**开发流程：**

```
需求分析 → 设计 → 编码 → 测试 → 部署 → 反馈
   ↑                                      ↓
   └──────────────────────────────────────┘
              迭代循环
```

**迭代计划：**

**第一迭代（基础功能）：**
- 用户注册登录
- 角色权限控制
- 基础页面框架

**第二迭代（核心功能）：**
- 单词学习功能
- 学习记录管理
- 任务管理系统

**第三迭代（高级功能）：**
- 班级管理
- 换班审核流程
- 错题本功能

**第四迭代（数据分析）：**
- 学习进度统计
- 数据可视化
- 教师数据看板

**第五迭代（优化完善）：**
- 性能优化
- 用户体验优化
- Bug 修复


### 1.3 前后端分离开发模式

**开发模式说明：**

前端和后端独立开发、独立部署，通过 RESTful API 进行数据交互。前端负责用户界面和交互逻辑，后端负责业务逻辑和数据管理。

**优势：**
- 前后端并行开发，提高开发效率
- 职责分离，降低耦合度
- 前端可以灵活选择技术栈
- 后端 API 可以被多个客户端复用（Web、移动端）

**接口约定：**

采用 RESTful API 设计规范：

```
资源操作规范：
GET    /api/users          # 获取用户列表
GET    /api/users/:id      # 获取指定用户
POST   /api/users          # 创建用户
PUT    /api/users/:id      # 更新用户
DELETE /api/users/:id      # 删除用户

统一响应格式：
{
  "code": 200,              // 状态码
  "message": "success",     // 消息
  "data": { ... }           // 数据
}
```

**开发协作流程：**

1. **接口设计阶段**：前后端共同制定 API 接口文档
2. **并行开发阶段**：
   - 前端：使用 Mock 数据进行开发
   - 后端：实现 API 接口
3. **联调测试阶段**：前端对接真实后端接口
4. **集成部署阶段**：前后端分别部署

```sql
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '用户ID',
```

```sql
  `is_mastered` BOOLEAN DEFAULT FALSE COMMENT '是否已掌握',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_word` (`user_id`, `word_id`),
  KEY `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='错题本表';

-- 换班申请表
CREATE TABLE `class_change_request` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '申请ID',
  `student_id` BIGINT NOT NULL COMMENT '学生ID',
  `from_class_id` BIGINT NOT NULL COMMENT '原班级ID',
  `to_class_id` BIGINT NOT NULL COMMENT '目标班级ID',
  `reason` TEXT COMMENT '申请理由',
  `status` ENUM('PENDING', 'APPROVED', 'REJECTED') DEFAULT 'PENDING' COMMENT '状态',
  `reviewer_id` BIGINT COMMENT '审核人ID',
  `review_comment` TEXT COMMENT '审核意见',
  `apply_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '申请时间',
  `review_time` DATETIME COMMENT '审核时间',
  PRIMARY KEY (`id`),
  KEY `idx_student_id` (`student_id`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='换班申请表';
```

**数据库索引优化：**
- 主键索引：每个表的 `id` 字段
- 唯一索引：`username`、`word`、`user_id + word_id` 组合
- 普通索引：`class_id`、`user_id`、`word_id`、`learn_time`、`deadline` 等高频查询字段

## 五、开发规范总结

### 5.1 代码规范

**后端代码规范：**
- 遵循阿里巴巴 Java 开发手册
- 类名使用大驼峰命名（UserService）
- 方法名使用小驼峰命名（getUserById）
- 常量使用全大写下划线分隔（MAX_SIZE）
- 每个方法添加注释说明
- 使用 Lombok 简化代码

**前端代码规范：**
- 遵循 Vue 3 官方风格指南
- 组件名使用 kebab-case（word-learn.vue）
- 变量名使用小驼峰命名（currentWord）
- 使用 ESLint 检查代码
- 使用 Prettier 格式化代码

### 5.2 Git 提交规范

- 提交信息使用中文或英文
- 遵循 feat/fix/docs 等前缀规范
- 每次提交只包含一个功能或修复
- 提交前先拉取最新代码
- 解决冲突后再提交

### 5.3 文档规范

- 使用 Markdown 编写文档
- 文档结构清晰，层次分明
- 代码示例完整可运行
- 及时更新文档内容

## 六、工具对比与选择

### 6.1 后端 IDE 对比

| 工具 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| IntelliJ IDEA | 功能强大、智能提示好、插件丰富 | 收费（社区版免费） | 企业级 Java 开发 |
| Eclipse | 免费开源、插件多 | 界面较旧、性能一般 | 个人学习、小型项目 |
| VS Code | 轻量级、启动快 | Java 支持不如 IDEA | 轻量级开发 |

**本项目选择：IntelliJ IDEA**
- 理由：功能最强大，开发效率最高，适合 Spring Boot 项目

### 6.2 前端编辑器对比

| 工具 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| VS Code | 免费、插件丰富、性能好 | 需要配置插件 | 前端开发首选 |
| WebStorm | 功能强大、开箱即用 | 收费、占用资源多 | 企业级前端开发 |
| Sublime Text | 轻量级、启动快 | 插件生态不如 VS Code | 简单编辑 |

**本项目选择：Visual Studio Code**
- 理由：免费、插件丰富、社区活跃、适合 Vue 开发

### 6.3 数据库工具对比

| 工具 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| MySQL Workbench | 官方工具、功能全面 | 界面较复杂 | 数据库设计、管理 |
| Navicat | 界面友好、功能强大 | 收费 | 专业数据库管理 |
| DBeaver | 免费开源、支持多种数据库 | 功能不如 Navicat | 个人开发 |

**本项目选择：MySQL Workbench**
- 理由：官方工具、免费、功能完整、适合数据库设计

## 七、总结

本项目采用面向对象的分析与设计方法，遵循 SOLID 原则和设计模式，使用前后端分离的开发模式。后端使用 IntelliJ IDEA 进行 Spring Boot 开发，前端使用 Visual Studio Code 进行 Vue 3 开发，通过 Git 进行版本控制，使用 MySQL Workbench 进行数据库设计和管理。整个开发过程遵循敏捷开发方法，采用迭代式开发，快速交付可用的软件增量。通过合理选择开发工具和规范的开发流程，保证了项目的开发效率和代码质量。
